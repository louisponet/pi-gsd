<objective>
Set up oclif alongside commander.js and migrate the five highest-traffic commands (state, init, roadmap, config, phase) to typed oclif command classes in src/commands/. Commander.js remains active during this transition period - both can coexist in src/cli.ts.

CRITICAL ARCHITECTURE NOTE: This project builds to a single CJS bundle via tsup. oclif's plugin-based autodiscovery (scanning a commands/ directory at runtime) does NOT work with bundled code. We use oclif's programmatic registration instead: every command class is imported explicitly in src/cli.ts and registered via run() directly. This gives us all the type-safety benefits of oclif without requiring dynamic file loading.
</objective>

<interfaces>
oclif command class pattern (non-plugin, explicit registration):
  ```typescript
  import { Command, Flags, Args } from "@oclif/core";

  export class StateJsonCommand extends Command {
    static description = "Output project state as JSON";
    static flags = {
      raw: Flags.boolean({ description: "Output raw JSON" }),
    };
    async run() {
      const { flags } = await this.parse(StateJsonCommand);
      // call lib/state.ts functions
    }
  }
  ```

Execution model: commands are NOT registered with oclif's plugin system.
Instead, src/cli.ts routes argv to the correct class and calls:
  ```typescript
  await CommandClass.run(remainingArgs);
  ```

This means oclif provides: typed flags/args, auto-generated help (--help), and class structure.
It does NOT provide: plugin autodiscovery, command tree auto-building from directory.

Help generation: each command class has static description, static flags, static args.
`pi-gsd-tools state json --help` is handled by passing `--help` to CommandClass.run().

Existing command signatures to preserve (from src/cli.ts):
  state json [--raw] [--pick <field>]
  state update <key> <value>
  state reset
  init plan-phase <phase> [--ws <workspace>]

  roadmap analyze [--raw]
  roadmap get-phase <phase> [--pick <field>]
  config-get <key>
  config-set <key> <value>
  phase list
  phase status <phase>
</interfaces>
